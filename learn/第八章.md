# 静态和动态渲染

在上一章中，你获取了 dashboard 概览页面的数据。然而，我们简要讨论了当前设置的两种局限性：

1. 数据请求正在造成一个无意的瀑布效应。
2. 仪表板是静态的，因此任何数据更新都不会反映在你的应用程序上。

- 什么是静态渲染，以及它如何提升你的应用性能。
- 什么是动态渲染以及何时使用。
- 不同法人方法让你的 dashboard 动态更新
- 模拟一个慢速的数据获取，看看会发生什么

## 什么是静态渲染？

静态渲染数据时，数据的获取和渲染是在构建时（部署时）或 [重新验证数据](https://nextjs.org/docs/app/building-your-application/data-fetching/fetching-caching-and-revalidating#revalidating-data) 时发生的。

每当用户访问你的应用程序时，缓存结果都会被送达。静态渲染有几个好处：

- 更快的网站--预渲染内容可以缓存并部署到 [Vercel](https://vercel.com/) 等平台时全球分发。这确保全球用户能够更快、更可靠地访问您的网站内容。
- 降低服务器负载--由于内容被缓存，服务器无需为每个用户请求动态生成内容。这可以降低计算成本。
- SEO--预渲染内容更容易被搜索引擎爬虫索引，因为页面加载时内容已经存在。这有助于提升搜索引擎排名。

静态渲染适用于 没有数据 或 用户间共享数据 的用户界面，比如静态博客文章或产品页面。它可能不适用于有个性化数据且经常更新的 dashboard。

静态渲染的反面是动态渲染。

## 什么是动态渲染？

动态渲染时，每个用户在 请求时(用户访问页面时) 在服务器上渲染内容，动态渲染有几个好处：

- 实时数据--动态渲染允许您的应用程序显示实时或频繁更新的数据。这非常适合数据频繁变化的应用。
- 用户特定内容--提供个性化内容(如 dashboard 或 用户资料)更为便捷，并根据用户互动更新数据。
- 请求时间信息--动态渲染允许您访问只能在请求时知道的信息，如 cookie 或 URL 搜索参数。

## 模拟慢速数据获取

我们正在构建的 dashboard 应用是动态的。

然而，上一章中仍有一个问题。如果数据请求比其他所有请求都慢，会发生什么？

让我们模拟一个慢速的数据取用。在 `app/lib/data.ts` 中，取消注释 `console.log` 并设置 `fetchRevenue()` 内的 `Timeout`：

```ts
// /app/lib/data.ts

export async function fetchRevenue() {
  try {
    // We artificially delay a response for demo purposes.
    // Don't do this in production :)
    console.log("Fetching revenue data...");
    await new Promise((resolve) => setTimeout(resolve, 3000));

    const data = await sql<Revenue[]>`SELECT * FROM revenue`;

    console.log("Data fetch completed after 3 seconds.");

    return data;
  } catch (error) {
    console.error("Database Error:", error);
    throw new Error("Failed to fetch revenue data.");
  }
}
```

现在在新标签页打开 [http://localhost:3000/dashboard/](http://localhost:3000/dashboard/)，注意到页面加载时间变长了。在你的终端中，你还应该会看到以下消息：

```txt
Fetching revenue data...
Data fetch completed after 3 seconds.
```

这里，你添加了一个人为的 3 秒延迟，模拟了慢速取数据。结果是，在数据获取过程中，你的整个页面都被阻挡，无法向访客展示 UI。这就引出了开发者必须解决的一个常见挑战：

在动态渲染中， **你的应用速度取决于你最慢的数据取用速度**。
